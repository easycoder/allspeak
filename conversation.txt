# How to makeEasyCoder multilingual

> EasyCoder is a high-level scripting language based on English. There's no reason in principle why there can't be variants, using the same core technology, but based on other human languages. The syntax and vocabulary would be different to a greater or lesser extent, but they would work exactly the same way. Can you suggest how we might best accommodate multiple language variants while avoiding unnecessary code duplication?

Best way: treat each human-language variant as a **surface syntax layer** that maps to one shared internal representation.

**Core Architecture**
1. Keep one canonical engine and one canonical intermediate form.
2. Parse every language variant into the same canonical AST/bytecode/command stream.
3. Execute only canonical form in runtime modules (`Core.js`, `Browser.js`, etc.), never language-specific forms.

**How To Avoid Duplication**
1. Split compiler into phases:
   1. `Lexer/phrase matcher per language`
   2. `Canonical normalizer` (maps variant tokens to canonical command IDs)
   3. `Shared semantic validator`
   4. `Shared code generator/runtime bridge`
2. Make language packs data-driven:
   1. Vocabulary aliases (`put`, `set`, `if`, `while`, etc.)
   2. Multi-word phrase patterns
   3. Inflection/synonym rules
   4. Localized error/help strings
3. Use command IDs internally, not words:
   1. Example idea: map any localized form of “set style” to `CMD_SET_STYLE`.
   2. Runtime only sees `CMD_SET_STYLE`, so no duplication downstream.
4. Keep plugins canonical:
   1. Plugin authors define canonical command schema once.
   2. Language packs provide only alias mappings for plugin keywords.

**Recommended Implementation Model**
1. Add a `language` declaration in script header or compiler option.
2. Load `languages/<locale>.json` (or `.js`) pack at compile time.
3. Pack contains:
   1. `keywords`
   2. `phrase templates`
   3. `reserved words`
   4. `diagnostics text`
4. Compiler resolves localized tokens to canonical grammar rules.
5. Existing runtime stays unchanged.

**Testing Strategy**
1. Golden parity tests:
   1. Same program in English and Spanish should compile to identical canonical output.
2. Cross-language behavior tests:
   1. Execute both and compare runtime state/events.
3. Backward compatibility suite:
   1. English remains reference baseline.

**Practical Rollout**
1. Start with one “near-English” variant to validate architecture.
2. Externalize all current English keywords into an `en` language pack first.
3. Add a second language pack without touching runtime behavior.
4. Expand docs/Codex to show side-by-side examples mapping to the same canonical command IDs.

This gives you multilingual syntax with one engine, one semantics layer, and minimal long-term maintenance cost.

--------------------------------------------------------------------------------
> I take it you're proposing the compiler deals primarily with meaning, rather than the specific syntax a given language uses to convey it.

Exactly.

The compiler should primarily operate on meaning (canonical operations), while each language pack handles how that meaning is expressed in natural syntax.

Think of it as:

Language-specific front end: “how it is said”
Shared semantic core: “what it means”
Shared runtime: “how it runs”
That separation is what gives you multilingual support without cloning the whole compiler/runtime per language.

--------------------------------------------------------------------------------
> Regarding making EasyCoder multilingual.
>
> Human languages can differ in the structure of sentences as well as the words used. For example:
> put {value} into {variable}
> might in another language or dialect be
> into {variable} place {value}
>
> The only things the compiler cares about are
> identifying {value} and {variable}
> identifying the purpose of the command - it's meaning, independent of language used.
>
> This is presumably the job of the lexer frontend, using separate translation packs for each language. Does the outline you previously gave take this into account, and permit additional languages to be added simply by enhancing the lexer?

Yes. The outline does take this into account, with one important refinement: for many languages, it is not only the lexer that changes, but the whole front end (`tokenizer + parser/normalizer`).

Your example is exactly right:

- English style: `put {value} into {variable}`
- Alternate style: `into {variable} place {value}`

Both should normalize to the same canonical internal command, for example:

- `op=put`
- `arg1=value`
- `arg2=variable`

Then the existing compiler/runtime stays unchanged.

So the scalable design is:

1. Keep one canonical intermediate representation (IR) and one backend.
2. Add a language pack that maps surface syntax to canonical IR.
3. Let each language pack define its own patterns/word order/morphology.
4. Reuse semantic checks, optimizer, and runtime after normalization.

Practical caveat:
- If a new language only changes keywords, lexer changes may be enough.
- If it changes word order, agreement, inflection, particles, or optional omissions, you also need parser/grammar rules in that language pack.
- So “enhancing the lexer” is directionally correct, but in practice think “enhancing the front-end language pack.”

That still satisfies your goal: adding languages without touching core compiler/runtime logic.

--------------------------------------------------------------------------------
> So do we then identify each 'meaning' with its own universal mnemonic?
Yes, exactly. That is the cleanest way to do it.

Use a language-neutral opcode (universal mnemonic) per semantic intent, for example:

- `ASSIGN` for “put X into Y”
- `IF`, `WHILE`, `CALL`, `RETURN`
- `ADD`, `SUB`, `COMPARE_EQ`

Each language pack maps local phrasing and word order into those same opcodes plus typed arguments.

So your pipeline becomes:

1. Surface text (any language)
2. Front-end parse/normalize
3. Canonical opcode + operands
4. Existing compiler/runtime

This gives you:
- Easy addition of new languages
- Shared backend behavior
- Consistent debugging and tooling across languages

One useful implementation detail is to keep an opcode table (stable IDs + arg schema) as the single source of truth.
